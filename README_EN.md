# 🤖 XuanShu v5.8.1

**A Local AI Agent Cluster System with Self-Evolution Capability**

XuanShu is more than just a chat interface. It is a cognitive system capable of self-reflection, self-upgrading, and LAN collaboration. It follows the "local-first" principle, ensuring all data stays within the domain, creating a true private digital brain.

## Main Target Users:
- **1. Designed for mid-to-low-end GPUs. LAN mode allows multiple home GPUs like 1080, 2080 to work together.**
- **2. New [Cross-Internet Remote Connection] support: Local GPU PC ↔ Remote GPU Server. No matter where you are, as long as TCP connection is available, you can form a distributed computing cluster!**
- **3. Devices with confidentiality requirements can deploy models locally, ensuring all data stays within the domain, absolutely leak-proof.**
- **4. Fully local, backdoor-free, open-source, and completely controllable. You can freely optimize and upgrade.**

## ✨ Core Features

- **🧠 Self-Evolution Cognition**: Built-in `EvolutionEngine` that learns from mistakes and automatically generates new Python skill plugins.
- **🌐 LAN Collaboration**: One-click room creation/joining with UDP triple-discovery mechanism ensuring 100% room discovery even in relay router scenarios.
- **🚀 Cross-Internet Remote Connection**: Manually specify IP/domain to break through LAN limitations! Local GPU PCs and remote GPU servers can connect across regions to form distributed computing clusters with heartbeat keep-alive, real-time load reporting, and collaborative task allocation!
- **🔄 Collaborative Mode Skill Sync**: [New in v5.7.0] In collaboration mode, new skills generated by any Agent are automatically synchronized to all nodes in the cluster, achieving "generate once, share globally"!
- **🎯 v5.8.0 MVP Final Release**: The major version 5 series officially concludes. Multi-machine collaboration Agent completes MVP with full LAN/cross-internet collaboration capabilities!
- **💾 Dual-Track Memory System**: Combines SQLite (high-performance subconscious) with Markdown (editable conscious) for precise long-term memory retrieval and human guidance.
- **🧠 Three-Level Degradation Vector Memory**: Vector semantic search → SQLite keyword matching → Pure JSON file storage. Automatically detects dependency availability, adapting from high-end servers to embedded devices.
- **👁️ Web Resource Direct Display**: Perfectly supports direct preview of model-generated images, audio, and video on the Web.

#### Cluster API Token Configuration

If you see `[CONFIG Security Warning] Default Cluster Token detected` during startup, you need to configure a permanent token.

**Configuration Method (Recommended before startup):**

**Windows Users (PowerShell):**
```powershell
# Temporary setting (current window only)
$env:CLUSTER_API_TOKEN = "your-secure-random-token-32-chars-min"
python launcher.py

# Permanent setting (user level)
[Environment]::SetEnvironmentVariable("CLUSTER_API_TOKEN", "your-secure-random-token-32-chars-min", "User")
# Requires reopening terminal to take effect
```

**Windows Users (CMD):**
```cmd
set CLUSTER_API_TOKEN=your-secure-random-token-32-chars-min
python launcher.py
```

**Linux / macOS Users:**
```bash
# Temporary setting (current terminal only)
export CLUSTER_API_TOKEN="your-secure-random-token-32-chars-min"
python launcher.py

# Permanent setting (write to ~/.bashrc or ~/.zshrc)
echo 'export CLUSTER_API_TOKEN="your-secure-random-token-32-chars-min"' >> ~/.bashrc
source ~/.bashrc
```

> **Token Requirements**: At least 32 characters recommended, containing letters, numbers, and symbols to ensure cluster communication security. All nodes joining the same cluster must use the same `CLUSTER_API_TOKEN`.

## 🗺️ Cluster Collaboration — Distributed Computing Power

XuanShu fully supports two modes of cluster collaboration: **LAN Auto-Discovery** + **Cross-Internet Remote Manual Connection**, allowing your local 1080/2080 GPUs to work perfectly with GPU servers anywhere.

### 🎯 Comparison of Two Collaboration Modes

| Mode | Use Case | Discovery Method | Cross-Region Support |
|------|----------|------------------|---------------------|
| **Mode A: LAN Auto-Discovery** | Multiple computers at home/office on the same network segment | UDP triple broadcast + intelligent unicast detection | ❌ LAN only |
| **Mode B: Manual IP Specification for Cross-Internet Connection** | Local home machine + cloud GPU server, distributed nodes in different locations | Direct TCP peer-to-peer connection | ✅ Anywhere in the world, as long as network is connected |

---

### 📖 Mode A: LAN Auto-Collaboration (100% Discovery Even with Relay Routers)

Even if devices behind relay routers are not visible in the main router's device list, XuanShu's **UDP Triple Guarantee Discovery Mechanism** can still find all rooms:
1. Unconditional global limited broadcast `255.255.255.255`
2. Auto-generate subnet broadcast addresses
3. Fallback to hundreds of commonly used IP unicast probes

#### Quick Start LAN Collaboration
1. All machines on the same WiFi/LAN
2. First machine "Create Room", automatically starts UDP broadcast
3. Other machines open room list, wait a few seconds for auto-discovery and join
4. Start collaborative work directly

---

### 🚀 Mode B: Cross-Internet Remote Connection — Local ↔ Remote GPU Server (Core New Feature)

This is the most powerful mode! No matter where your remote GPU server is - in the cloud, at an offsite office - as long as TCP network is available, you can form a distributed cluster.

#### Usage Example: "Local 1080 Home PC + Cloud GPU Server Collaborative Work"

| Step | Host Side (Remote GPU Server) | Member Side (Local Home GPU PC) |
|------|-------------------------------|---------------------------------|
| 1️⃣ Preparation | Ensure you can connect to cloud server via public IP/SSH, firewall/cloud provider security group allows **port 30001 (TCP)** | Run XuanShu normally locally, no additional port mapping needed |
| 2️⃣ Create Room | Run XuanShu, "Create Room" normally, set alias and model | Open XuanShu main interface |
| 3️⃣ Call Manual Join API | — | Send POST request to `/api/cluster/rooms/manual-join` |
| 4️⃣ Fill Request Body | — | `{ "host_ip": "your cloud server public IP or domain", "alias_name": "your local alias", "model": "your local model name like qwen2.5:7b", "password": "room password set by host (empty string if none)" }` |
| 5️⃣ Complete | Your local node is online in host's member list! Heartbeat keep-alive runs automatically, load reports in real-time | Connection successful, joined collaboration, host can assign tasks you're good at locally |

#### API Interface
- `POST /api/cluster/rooms/manual-join` — Core interface, manually specify remote host IP to join
- `POST /api/cluster/rooms/manual-leave` — Safely disconnect remote connection
- `GET /api/cluster/discovery/local-rooms` — Get list of all LAN rooms discovered via UDP

In collaboration mode, new skills generated by any Agent can be **automatically synchronized** to all other nodes in the cluster, achieving "generate once, share globally".

#### Automatic Synchronization Mechanism

When any Agent in the cluster (host or Worker) generates a new skill:
1. Skill code is verified locally and saved to `skills/auto_generated/<skill_name>/` directory
2. System automatically generates corresponding `SKILL.md` file containing trigger scenarios, parameter descriptions, return value descriptions, etc.
3. System automatically triggers `broadcast_skill_sync()` to broadcast to all online nodes
4. Each node receives and saves to `skills/auto_generated/synced_<skill_name>_timestamp/` directory and automatically registers
5. All nodes can immediately use this new skill

### 🌐 Cross-Platform Port Opening Instructions

To enable normal discovery and connection to collaboration rooms among multiple devices on LAN, you need to ensure the following key ports are open in the firewall:

| Port | Protocol | Purpose | Description |
|------|----------|---------|-------------|
| **30000** | TCP | Web interface service | Default port for all users to access XuanShu interface via browser |
| **30001** | TCP | Cluster Manager listening | Port for all Worker nodes to connect to host via TCP for task distribution |
| **50005** | UDP | LAN room discovery broadcast | UDP protocol for scanning and discovering collaboration room information on the same network segment |

#### 🪟 Windows Users

**⚠️ Required Settings for LAN Collaboration**

To allow other LAN devices to discover and connect to your XuanShu collaboration room, you need to complete the following two settings:

**1. Enable Network Discovery and File Sharing**

Open "Control Panel - Network and Sharing Center - Advanced Sharing Settings", ensure the following options are enabled:
- **Enable Network Discovery**: Allow other devices to discover your computer on the network
- **Enable File and Printer Sharing**: Allow devices on the network to access your shared resources

**2. Windows Defender Firewall Port Exemption**

Run the following PowerShell commands (as administrator):
```powershell
# Run PowerShell as administrator
New-NetFirewallRule -DisplayName "XuanShu 30000" -Direction Inbound -Protocol TCP -LocalPort 30000 -Action Allow
New-NetFirewallRule -DisplayName "XuanShu 30001" -Direction Inbound -Protocol TCP -LocalPort 30001 -Action Allow
New-NetFirewallRule -DisplayName "XuanShu 50005" -Direction Inbound -Protocol UDP -LocalPort 50005 -Action Allow
```

> **Important Note**: The above settings are necessary conditions for normal collaborative tasks. If network discovery is not enabled and ports are not allowed, other devices will not be able to discover your collaboration room or establish TCP connections for task allocation, resulting in inability to perform collaborative tasks effectively.

#### 🍎 macOS Users
In "System Settings - Network - Firewall", manually add Python program to allow incoming connections, or use the following command to temporarily allow:
```bash
sudo pfctl -e  # Enable pf firewall and configure rules to allow 30000/30001(UDP50005)
```

#### 🐧 Linux Users (Ubuntu/Debian/CentOS General)
```bash
# Ubuntu/Debian using ufw
sudo ufw allow 30000/tcp
sudo ufw allow 30001/tcp
sudo ufw allow 50005/udp
sudo ufw reload

# CentOS/RHEL using firewalld
sudo firewall-cmd --add-port=30000/tcp --permanent
sudo firewall-cmd --add-port=30001/tcp --permanent
sudo firewall-cmd --add-port=50005/udp --permanent
sudo firewall-cmd --reload
```
