# 玄枢复盘机制

> 所谓"上善若水"，复盘如水 —— 事后反思，方能顺势而为。

---

## 📖 核心理念

复盘是修行的重要组成部分。玄枢的复盘机制遵循以下原则：

1. **事后必复** - 每次重大操作或错误后必须复盘
2. **实事求是** - 如实记录，不夸大不缩小
3. **持续改进** - 复盘结果要落实到具体行动

---

## 🎯 何时复盘

### 自动触发
- 代码修改后测试发现Bug
- 新功能上线后出现预期外行为
- 用户反馈问题

### 手动触发
- 完成重要功能模块
- 进行架构调整
- 解决复杂技术问题
- 定期（如每周）代码审查

---

## 📋 复盘流程

### 1. 事实记录
- 时间、地点、参与者
- 发生了什么（客观描述）
- 预期是什么
- 实际结果是什么

### 2. 根因分析
- 为什么发生？
- 直接原因是什么？
- 根本原因是什么？
- 是否有模式/规律？

### 3. 经验提炼
- 这次学到了什么？
- 可以总结成什么原则？
- 如何防止类似问题？

### 4. 行动落实
- 具体改什么？
- 谁来改？
- 什么时候完成？

## 🎓 复盘案例

### [2026-05-08] 模型选择不一致问题

**背景**: 用户在launcher.py中选择了一个OpenAI兼容模型，但打开Web端后发现显示的是另一个模型。

**根因**: `/api/models`端点的过滤逻辑排除了没有api_key的OpenAI兼容模型。

**教训**:
1. 配置管理的过滤条件需要仔细检查
2. 不同入口（CLI/Web）应使用统一的配置源

**行动**:
- [x] 移除不合理的过滤条件
- [ ] 增加配置一致性测试
