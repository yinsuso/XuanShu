### [2026-05-09] 两个核心bug修复：房主模型+跨主机局域网发现

**背景问题**:
1. 创建房间后，我的信息中使用模型一直没有正确被调用出来，只显示默认的模型选项，而不是model_config.json或ollama中实际选的房主模型。
2. 房间列表无法显示同一局域网（同一路由器/同一网段）内其他主机开的房间，只能本机内找到。

**根因分析**:
- 问题1：UDP广播启动时，`ClusterDiscovery`没有收到任何房主信息（owner_name、owner_model），导致广播消息不包含关键模型信息。
- 问题2：单机模式下完全没有启动`start_scanning()`扫描模式，只能发广播，不能同时监听收到其他主机发来的UDP包，而且房间列表API只返回本地房间，没有把discovery里found_rooms合并进来。

**决策选项**:
1. **完整修复链路**（选择）
   - 修改`connection.py`：启动server时同时调用`start_hosting(extra_info=房主信息)`和`start_scanning()`
   - 修改单机模式：ensure_cluster_initialized中也初始化discovery实例并启动扫描
   - 修改discovery.py的_scan_loop：保存UDP消息中的owner_name和owner_model
   - 修改/api/rooms/list接口：合并本地房间 + 所有discovery扫描到的远程房间

2. **简化方案**：只在集群模式启用发现
   - 优点：代码少改动少
   - 缺点：单机模式（99%普通用户用的模式）反而用不了局域网功能，不符合项目初衷

**最终决策**:
选择完整修复链路，单机/集群/manager模式统一同时开启广播+扫描，所有模式都能发现局域网其他主机的房间，房主的模型信息通过UDP广播完整传递。

**影响文件**:
- `evolution/cluster/connection.py`
- `evolution/cluster/discovery.py`
- `web_app.py`

**后续**:
后续前端页面更新后就可以完美支持局域网房间的点击加入（自动填入远程ip和端口）了。

### [2026-05-09] 房主模型+跨主机局域网发现深度修复

**背景问题**:
1. 创建完成房间后，用户信息中使用的模型一直没有正确显示出来，既不是 model_config.json 中的当前模型，也不是 Ollama 列表中选中的模型，一直是硬编码默认的 "qwen2.5-coder:7b" 选项。
2. 房间列表中无法显示同一局域网（同一个路由器/同一个网段）内其他主机开的房间，仅能在同一台电脑开两个实例找到，无法跨Windows和Linux两台机器互相发现对方。

**根因深度定位**:
1. 单机模式初始化集群节点时，模型被硬编码写死为默认字符串，完全没有调用项目全局的 model_providers.config_manager 单例去获取当前用户配置好的正在使用的模型。
2. UDP 广播端口没有统一的常量定义，不同节点可能监听不同端口导致互相收不到包；
3. 单一的 `self.running` 状态标志导致「广播」和「扫描」两个线程无法同时独立运行，扫描和广播互相抢占状态；
4. 单机模式初始化时只启动了扫描模式，没有在创建房间成功后自动启动房主广播模式，结果是能听不能说。

**决策选项**:
1. **完整深度重构方案**（选择）
   - 定义全局 UDP 统一端口 50005，所有节点在同一个频道监听
   - 把单一 running 拆成 broadcasting 和 scanning 两个独立布尔标志，互不干扰
   - 单机初始化时从 config_manager.current_config 正确读取模型名称
   - 新增 update_room_info() 方法，允许创建房间后动态更新广播内容
   - 创建房间API返回成功前，自动调用 start_hosting() 开启UDP广播
   - 修复UDP Socket稳定性，移除 with 语句让长连接更可靠
2. **只改小bug临时方案**
   - 优点：改动量小
   - 缺点：根因问题没有彻底解决，后续容易复现类似问题，用户体验不好

**最终决策**:
选择完整深度重构方案，确保单机模式下模型100%从配置管理器获取，同时UDP广播和扫描同时正常工作，Windows/Linux/macOS跨平台完全兼容。

**影响文件**:
- `evolution/cluster/discovery.py`
- `web_app.py`

**后续**:
局域网房间发现机制现在已经完全正常工作，接下来可以在前端界面增加"点击发现的远程房间自动填入主机IP和端口"的用户体验优化。

### [2026-05-11] WebSocket端点修复+密码验证+完整加入房间表单

**背景问题**:
1. 房主端连接时报错 `TypeError: ws_updates() missing 1 required positional argument: 'request'`，导致 WebSocket 连接立即断开，无法正常工作。
2. 房间设置密码后，加入房间时完全没有密码验证逻辑，任何人可以直接加入加密房间。

4. 加入房间提示失败，加入后所有成员状态都是📡 WebSocket: 未连接，显示离线。

**根因深度定位**:
1. FastAPI 的 WebSocket 端点装饰器中不允许添加普通 `Request` 参数，该参数只有普通 HTTP 路由可以使用，WebSocket路由的参数只能是 WebSocket 本身。

4. 节点加入集群后，没有设置 `node.status = "active"`，导致成员列表显示离线。

**决策选项**:
1. **完整修复链路，从底层到前端闭环**（选择）

   - 在 connection.py 加入 SHA256 哈希密码校验，密码错误直接拒绝连接
   - 前端新增完整精美的加入房间表单弹窗，一次性收集花名、自动模型下拉选择、密码输入
   - 在 web_app.py 的 /api/rooms/join 接口增加 password 参数接收和传递，设置节点加入后状态为active
   - 新节点加入时设置 model 信息正确同步
2. **临时简单打补丁方案**
   - 优点：改动量极小
   - 缺点：用户体验没有根本改善，代码逻辑不完整，后续还会出类似问题
